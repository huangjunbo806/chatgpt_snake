# Docker Snake 全栈学习项目设计规格

## 1. 项目背景

`docker_snake` 是一个面向 Web 零基础学习者的全栈项目。项目以网页贪吃蛇为主线，逐步讲解 HTML、CSS、JavaScript、HTTP、Express、PostgreSQL、Git、GitHub 和 Docker。

最终产品允许访客直接游玩。用户也可以使用用户名和密码注册、登录、提交个人最高分并查看全服排行榜。开发者将一个自定义应用镜像上传至火山引擎镜像仓库（Container Registry，CR），再由自己的服务器拉取并运行。

项目的首要目标是帮助学习者理解每一层如何工作，而不是用最少代码快速堆出成品。因此，功能按可运行里程碑逐步增加，代码保持职责清晰，配套中文课程、自动测试和可追溯的 Git 历史。

## 2. 学习目标

完成项目后，学习者应能解释并实践：

- HTML 如何描述网页结构。
- CSS 如何控制布局、响应式适配和视觉样式。
- JavaScript 如何响应输入、更新状态并使用 Canvas 绘图。
- 浏览器如何通过 HTTP 和 JSON 调用后端 API。
- Express 如何提供静态文件、路由、中间件和错误处理。
- PostgreSQL 如何保存用户、Session 和最高分。
- 密码哈希、Cookie、Session、CSRF 和限流的基本作用。
- Git 如何记录变更，GitHub 如何管理远程仓库、分支、Issue 和 Pull Request。
- Docker 镜像、容器、网络、Volume 和 Compose 分别解决什么问题。
- 源代码仓库、容器镜像仓库和持久化数据为何必须分开管理。
- 如何构建、推送、拉取、升级和回滚一个服务器应用。

## 3. 产品目标

- 提供终端霓虹风格的响应式网页贪吃蛇。
- 支持桌面键盘和移动端触控。
- 支持访客直接游玩。
- 访客最高分只保存在当前浏览器，不进入全服排行榜。
- 支持用户名和密码注册、登录及退出。
- 每名用户只在数据库保存一个历史最高分。
- 显示全服前 100 名以及当前用户自己的名次。
- 使用休闲级分数校验和请求限流，明确说明其防作弊边界。
- 使用单个自定义应用镜像同时提供前端和 API。
- 使用 PostgreSQL 官方容器及 Docker Volume 持久化数据。
- 提供从本地开发到火山引擎 CR 部署的中文教程。

## 4. 非目标

第一版不实现：

- 邮箱、手机验证码或第三方账号登录。
- 忘记密码、邮件验证或密码重置。
- 多因素认证（MFA）。
- 管理员后台和用户封禁界面。
- 单局成绩历史、周榜、月榜或赛季榜。
- 服务端实时游戏判定或操作回放验证。
- 可抵御专业攻击者的竞技级反作弊。
- 多实例横向扩展、Redis、消息队列或 Kubernetes。
- 多人实时游戏、障碍物、音效和皮肤系统。
- 自动购买域名、ICP备案或自动创建云服务器。
- 未经用户明确授权创建 GitHub 远程仓库或推送外部服务。

## 5. 渐进式课程里程碑

每个里程碑必须保持项目可运行，并在 `docs/lessons/` 提供对应教程。

| 里程碑 | 主题 | 可观察结果 |

| `lesson-03` | JavaScript | 贪吃蛇可以通过键盘和触控游玩 |
| `lesson-04` | HTTP 与 Express | Node.js 服务器同时提供网页和示例 API |
| `lesson-05` | PostgreSQL | 应用可以连接数据库并执行迁移 |
| `lesson-06` | 注册与登录 | 用户可以安全注册、登录和退出 |
| `lesson-07` | 排行榜 | 登录用户可以提交最高分并查看排名 |
| `lesson-08` | Docker | Compose 可以启动应用、数据库和 Volume |

## 6. 总体架构

```text
用户浏览器
  ├── 下载 HTML、CSS 和 JavaScript
  ├── 在本地运行贪吃蛇逻辑
  ├── 注册、登录和退出
  └── 提交、查看最高分
                 ↓ 同源 HTTP/JSON

Node.js + Express 应用容器
  ├── 提供前端 dist/ 静态文件
  ├── 提供 /api/* 接口
  ├── 校验用户输入和登录状态
  ├── 管理服务器端 Session
  └── 读取、更新排行榜
                 ↓ 参数化 SQL

PostgreSQL 18 容器
  ├── users 表
  ├── Session 表
  └── schema_migrations 表
                 ↓

Docker 命名 Volume
  └── 数据库持久化文件
```

浏览器和 API 使用同一个来源（Origin），不开放跨域 API。游戏移动、碰撞和绘图仍在浏览器运行；账号、Session 和最高分由服务器管理。

## 7. 仓库、镜像和数据边界

| 存储位置 | 保存内容 | 不保存内容 |
|---|---|---|
| Git/GitHub | 源码、测试、文档、Dockerfile、工作流 | 密码、`.env`、数据库文件 |
| 火山引擎 CR | 版本化应用镜像 | PostgreSQL 数据、Git 历史 |
| Docker Volume | PostgreSQL 持久化数据 | 应用源码和镜像 |
| 服务器环境配置 | 数据库密码、Session 密钥、部署参数 | 可公开提交的代码 |

升级或删除应用容器不得删除 PostgreSQL Volume。数据库备份使用 `pg_dump`，不能把运行中的数据库目录复制进应用镜像。

## 8. 技术栈

### 8.1 前端

- 原生 HTML：学习语义化页面结构和表单。
- 原生 CSS：实现终端霓虹主题、布局和响应式适配。
- 原生 JavaScript：实现游戏、DOM 交互和 API 请求。
- HTML Canvas：绘制棋盘、蛇和食物。
- Vite：在后期课程中提供开发服务器和生产资源打包，不引入 React 或 Vue。

### 8.2 后端

- Node.js 24 LTS：运行 JavaScript 服务端代码。
- Express：提供静态文件、API、中间件和统一错误处理。
- `pg`：使用参数化 SQL 访问 PostgreSQL。
- `argon2`：使用 Argon2id 计算和验证密码哈希。
- `express-session`：管理登录 Session。
- PostgreSQL Session Store：在数据库中保存 Session，生产环境不使用默认 `MemoryStore`。

### 8.3 数据库与工具

- PostgreSQL 18：保存用户、Session、迁移状态和最高分。
- Vitest：执行浏览器逻辑与服务器模块测试。
- Supertest：测试 Express API。
- Docker 与 Docker Compose：构建并编排应用和数据库。
- Git 与 GitHub Actions：版本控制和持续集成（CI）。

## 9. 游戏规则

### 9.1 棋盘与初始状态

- 棋盘为 `20 × 20` 网格。
- 蛇的初始长度为 4，初始方向向右。
- 棋盘上同时存在 1 个食物。
- 初始移动间隔为 140 ms。
- 最高理论分数为 `(400 - 4) × 10 = 3960`。

### 9.2 移动与输入

- 支持方向键和 WASD。
- 小屏设备显示触控方向键。
- 同一游戏步长只接受一次有效转向。
- 禁止从当前方向直接转向相反方向。
- 页面进入后台时自动暂停。
- 支持开始、暂停、继续和重新开始。

### 9.3 得分与加速

- 每吃到 1 个食物增加 10 分。
- 每吃到 5 个食物提升 1 个速度等级。
- 每级将移动间隔减少 10 ms。
- 最低移动间隔为 70 ms，达到后不再加速。

### 9.4 结束与胜利

- 撞墙或撞到自己时游戏结束。
- 不使用穿墙规则。
- 棋盘没有空闲单元格时进入胜利状态。
- 登录用户结束游戏后自动尝试提交成绩。
- 提交失败时保留当前成绩并提供重试按钮；刷新页面后不保证保留未提交成绩。

## 10. 界面设计

### 10.1 视觉方向

采用已确认的「终端霓虹」风格：

- 深绿色至黑色背景。
- 低对比度网格线。
- 绿色发光蛇身。
- 红色或粉色发光食物。
- 等宽字体与终端状态文字。
- 发光效果保持克制，保证文字可读性。

### 10.2 页面布局

采用已确认的「聚焦控制台」布局：

1. 顶部栏显示项目名称和当前登录状态。
2. 未登录时显示「注册」「登录」入口；登录后显示用户名和「退出」。
3. 游戏状态区显示当前分数、个人最高分和速度等级。
4. 中部显示响应式正方形 Canvas 棋盘。
5. 底部显示暂停、重新开始、提交重试和操作说明。
6. 小屏设备显示触控方向键。
7. 排行榜位于游戏控制台下方，显示前 100 名和「我的排名」。

注册和登录使用网页表单，不使用浏览器原生 `prompt()`。表单显示字段说明、校验结果和「第一版无法找回密码」提示。

### 10.3 可用性要求

- 游戏可通过键盘或触控完成全部操作。
- Canvas 保持正方形，不产生横向滚动。
- 表单具有可见标签，不只使用 placeholder。
- 按钮具有焦点、禁用和加载状态。
- API 错误显示为可理解的中文消息。
- Canvas 状态同时以普通文本表达，避免信息只依赖颜色或画面。
- 浏览器不支持 Canvas 时显示明确说明。

## 11. 用户流程

### 11.1 访客游玩

1. 打开网站。
2. 无需登录即可开始游戏。
3. 游戏结束后显示本局分数。
4. 浏览器使用 `localStorage` 保存访客最高分。
5. 未登录时提示「登录后完成的新游戏可进入排行榜」。

### 11.2 注册

1. 用户输入用户名、密码和确认密码。
2. 前端执行基础格式检查。
3. API 再次执行全部校验。
4. 后端计算 Argon2id 密码哈希并创建用户。
5. 注册成功后创建登录 Session，用户无需再次登录。

### 11.3 登录与退出

1. 登录成功后重新生成 Session ID。
2. 页面加载当前用户、个人最高分和排名。
3. 退出时销毁服务器 Session 并清除 Cookie。

访客本地最高分不会在登录时自动上传。登录后的新游戏才可以提交到服务器排行榜，避免把任意本地存储值直接当作可信成绩。

### 11.4 提交最高分

1. 游戏结束后，已登录客户端提交分数与本局持续时间。
2. API 校验 Session、数据格式、分数范围和请求频率。
3. 只有新分数严格高于数据库最高分时才更新。
4. 更新成功后返回新的最高分与名次。
5. 页面刷新排行榜。

## 12. 前端模块边界

### 12.1 游戏引擎

纯 JavaScript 游戏引擎不读取 DOM、Canvas、网络或存储。职责包括：

- 创建初始状态。
- 验证方向变化。
- 推进游戏步长。
- 处理食物、成长、计分和加速。
- 检测边界和自身碰撞。
- 从空闲单元格生成食物。
- 判断结束和胜利。

随机数生成器通过参数注入，使测试结果可重复。

### 12.2 Canvas 渲染器

渲染器只读取游戏状态并绘制背景、网格、蛇、食物和状态提示，不修改游戏状态。

### 12.3 输入控制器

输入控制器把键盘和触控输入转换为统一方向，并处理暂停、继续和重新开始。触控按钮只在自身区域阻止默认滚动行为。

### 12.4 访客最高分存储

访客最高分模块封装 `localStorage`。存储不可用或内容非法时回退到当前页面内存值，不影响游戏运行。该模块不保存密码、Session ID 或已登录用户的服务器最高分。

### 12.5 API 客户端

API 客户端统一处理：

- JSON 请求与响应。
- 同源 Cookie。
- 修改请求所需的自定义 CSRF 防护请求头。
- HTTP 状态码到用户消息的映射。
- 网络异常和加载状态。

### 12.6 页面控制器

页面控制器连接游戏、认证表单、排行榜和 API 客户端。它不包含密码哈希或 SQL 逻辑。

## 13. 后端模块边界

### 13.1 应用装配

应用装配模块创建 Express 实例并注册安全中间件、Session、静态文件、路由、404 和统一错误处理器。

### 13.2 认证服务

认证服务负责：

- 规范化并校验用户名。
- 校验密码长度。
- 计算和验证 Argon2id 哈希。
- 创建用户。
- 重新生成和销毁 Session。

### 13.3 排行榜服务

排行榜服务负责：

- 校验提交分数。
- 原子地执行「只提高、不降低」更新。
- 查询前 100 名。
- 查询当前用户排名。
- 处理同分排序规则。

### 13.4 数据访问层

数据访问层集中保存 SQL，并且只使用参数化查询。路由不得直接拼接 SQL 字符串。

### 13.5 迁移运行器

应用启动、监听 HTTP 端口之前执行版本化迁移：

- `schema_migrations` 记录已执行文件。
- 每个迁移只执行一次。
- 迁移失败时应用直接退出，避免在错误表结构上提供服务。
- 数据库必须先通过健康检查。

该自动迁移方案只面向单实例学习项目。多实例生产系统应改用独立迁移任务。

## 14. 数据模型

### 14.1 `users`

| 字段 | 类型 | 约束与说明 |
|---|---|---|
| `id` | `bigserial` | 主键 |
| `username` | `varchar(20)` | 唯一、只保存规范化小写值 |
| `password_hash` | `text` | Argon2id 哈希 |
| `best_score` | `integer` | 默认 0，范围 0～3960 |
| `best_score_at` | `timestamptz` | 首次达到当前最高分的时间；0 分时为空 |
| `created_at` | `timestamptz` | 默认当前时间 |

排行榜索引按 `best_score DESC, best_score_at ASC, id ASC` 排序。0 分用户不进入前 100 名，但登录后仍可看到「尚未上榜」。

### 14.2 Session 表

Session 表由 PostgreSQL Session Store 管理。Cookie 只保存无业务含义的 Session ID；用户 ID 和过期时间保存在服务器数据库中。

### 14.3 `schema_migrations`

| 字段 | 作用 |
|---|---|
| `version` | 唯一迁移版本 |
| `applied_at` | 执行时间 |

第一版不创建单局成绩表，因为产品只要求保存每名用户的历史最高分。

## 15. HTTP API

所有 API 使用 `/api` 前缀和 JSON。成功响应使用 `data` 字段；失败响应使用稳定的错误码与中文消息，不返回内部堆栈。

| 方法与路径 | 登录要求 | 作用 | 常见状态码 |
|---|---|---|---|
| `POST /api/auth/register` | 否 | 创建用户并登录 | 201、400、409、429 |
| `POST /api/auth/login` | 否 | 验证凭据并登录 | 200、400、401、429 |
| `POST /api/auth/logout` | 是 | 销毁 Session | 204、401 |
| `GET /api/auth/me` | 否 | 返回当前用户或访客状态 | 200 |
| `POST /api/scores` | 是 | 提交并尝试提高最高分 | 200、400、401、429 |
| `GET /api/leaderboard` | 否 | 返回前 100 名和当前用户名次 | 200 |
| `GET /healthz` | 否 | 检查应用和数据库 | 200、503 |

### 15.1 注册输入

- 用户名规范化后匹配 `^[a-z0-9_]{3,20}$`。
- 密码长度为 15～128 个 Unicode 字符。
- 服务端不接收或保存「确认密码」字段；确认密码只用于浏览器表单校验。

### 15.2 分数输入

- `score` 必须是 0～3960 之间且能被 10 整除的整数。
- `durationMs` 必须是 0～86,400,000 之间的整数。
- 正分数必须满足 `durationMs >= (score / 10) × 70`，该下限只拦截明显异常值。
- API 返回 `updated`、`bestScore` 和 `rank`。

## 16. 排行规则与休闲防作弊

- 每名用户只有一个排行榜位置。
- 只有严格更高的分数更新数据库。
- 同分时，较早达到该分数的用户排在前面。
- 每次更新使用一条带条件的原子 SQL，避免并发请求降低或覆盖最高分。
- 登录用户每分钟最多提交 12 次分数。
- 后端检查分数类型、范围、10 的倍数、持续时间和 Session。
- 异常提交记录必要的安全日志，但不记录密码或 Session ID。

由于游戏在浏览器运行，熟悉开发者工具的用户仍然可能伪造校验范围内的成绩。第一版把排行榜定位为学习和娱乐功能，不声称具有竞技公平性。未来可增加操作回放或服务端权威判定。

## 17. 认证与安全设计

### 17.1 用户名与密码

- 用户名去除首尾空白并转为小写。
- 只允许小写英文字母、数字和下划线，避免第一版处理 Unicode 同形字符。
- 密码允许 Unicode、空格和粘贴，不要求大小写、数字或特殊字符组合。
- 密码不静默截断。
- 使用 Argon2id 和独立随机盐，最低参数为 19,456 KiB 内存、2 次迭代、并行度 1。
- 数据库和日志永不保存明文密码。
- 第一版没有邮箱，因此无法自助找回密码；注册页面必须明确提示。

### 17.2 Session Cookie

- 使用框架生成的高熵 Session ID，不自行设计登录令牌。
- Session 数据保存在 PostgreSQL，不使用生产不适用的 `MemoryStore`。
- 登录成功后重新生成 Session ID，防止 Session 固定攻击。
- Cookie 使用不暴露技术的通用名称。
- Cookie 设置 `HttpOnly`、`SameSite=Lax` 和登录后 7 天的绝对有效期，不滚动续期。
- 生产环境通过 HTTPS 设置 `Secure`。
- 退出时销毁服务器 Session 并清除 Cookie。
- 登录状态不存入 `localStorage`。

### 17.3 CSRF 与同源策略

- 不启用宽泛 CORS，浏览器和 API 保持同源。
- 修改数据的请求只接受 `application/json`。
- 前端添加 `X-Docker-Snake-Request: 1`，后端缺少或值不匹配时拒绝。
- 后端验证 `Origin`，生产目标来源通过环境变量明确配置。
- 所有状态修改使用 `POST`，`GET` 不修改服务器数据。
- `SameSite=Lax` 只作为纵深防御，不作为唯一 CSRF 措施。

### 17.4 其他防护

- 使用安全响应头并隐藏不必要的技术指纹。
- JSON 请求体上限为 16 KiB。
- 所有输入在服务端再次验证。
- 注册按 IP 限制为每小时 5 次。
- 登录按 IP 限制为每 15 分钟 10 次失败尝试，并按规范化用户名限制为每 15 分钟 5 次失败尝试；登录成功后清除对应用户名计数。
- 登录错误统一返回「用户名或密码错误」。
- 使用参数化 SQL 防止 SQL 注入。
- 生产环境只允许 HTTPS；HTTP 仅用于 `localhost` 学习。
- 代理后的生产部署显式配置可信代理数量，不盲目信任任意转发头。

第一版限流使用单实例内存存储。若未来扩展多个应用实例，必须改用共享限流存储。

## 18. 异常处理

### 18.1 浏览器

- Canvas 不可用：停止游戏初始化并显示说明。
- API 网络失败：显示可重试消息，不让页面崩溃。
- 登录失效：切换为访客状态并提示重新登录。
- 分数提交失败：保留当前页面中的成绩并显示重试按钮。
- 快速连续转向：一个游戏步长只采用第一个合法转向。
- 页面进入后台：自动暂停并清理活动计时任务。

### 18.2 API

| 场景 | HTTP 状态码 |
|---|---|
| 请求格式或字段非法 | 400 |
| 用户名或密码错误 | 401 |
| 未登录访问受保护接口 | 401 |
| 用户名已存在 | 409 |
| 请求过快 | 429 |
| 数据库暂时不可用 | 503 |
| 未预期服务器错误 | 500 |

统一错误处理器生成请求 ID、记录必要上下文并返回稳定错误结构。响应不包含堆栈、SQL、环境变量、密码哈希或数据库连接串。

### 18.3 数据库与启动

- PostgreSQL 未就绪：应用每 2 秒重试一次，最多 30 次，随后退出并交给 Docker 重启策略接管。
- 迁移失败：应用不监听端口并退出非零状态。
- Session Store 不可用：认证请求失败，不降级为不安全的内存 Session。
- 健康检查查询失败：`/healthz` 返回 503，容器标记为不健康。

## 19. 构建与 Docker 设计

### 19.1 前端构建

Vite 只负责打包原生 HTML、CSS 和 JavaScript：

- 开发阶段提供热更新。
- 开发阶段由 Vite 把 `/api` 和 `/healthz` 代理到本地 Express 端口 3000，使浏览器仍按同源方式调用。
- 生产阶段输出带内容哈希的 `dist/` 静态资源。
- 默认不生成生产源码映射。
- Express 在生产环境提供 `dist/`。

### 19.2 应用镜像

Dockerfile 使用 Node.js 24 Alpine 多阶段构建。

构建阶段：

1. 复制 `package.json` 和锁文件。
2. 使用锁文件安装全部依赖。
3. 复制源码和测试。
4. 运行单元测试与前端构建。

运行阶段：

1. 安装生产依赖。
2. 复制服务器代码、迁移文件和 `dist/`。
3. 使用 Node 镜像内置的非 root 用户运行。
4. 监听容器端口 3000。
5. 提供 Docker `HEALTHCHECK`。

最终应用镜像不包含开发依赖、测试缓存、Git 元数据、`.env` 或数据库文件。

### 19.3 Compose 服务

`compose.yaml` 定义：

- `app`：构建或拉取自定义应用镜像。
- `db`：使用 PostgreSQL 官方镜像 `postgres:18.4-alpine`。
- `db-data`：挂载到 PostgreSQL 18 官方镜像规定的 `/var/lib/postgresql`。
- 私有默认网络：应用通过服务名 `db` 连接数据库。
- 数据库健康检查：应用等待数据库就绪。
- 应用健康检查：检查 `/healthz`。
- 重启策略：服务器异常重启后恢复服务。

默认只向宿主机发布应用端口 8080，数据库端口不公开。学习者通过 `docker compose exec db psql` 进入数据库。

### 19.4 环境配置

`.env.example` 只包含示例键名和安全说明：

- `POSTGRES_DB`
- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `DATABASE_URL`
- `SESSION_SECRET`
- `PUBLIC_ORIGIN`
- `TRUST_PROXY`
- `NODE_ENV`

真实 `.env` 必须加入 `.gitignore`。生产密钥使用安全随机值，不沿用示例文本。
`SESSION_SECRET` 解码后的随机内容不得少于 32 字节，数据库密码也必须与示例值不同。

## 20. Git 与 GitHub 设计

### 20.1 本地 Git

- 仓库默认分支为 `main`。
- 每个提交只完成一个清晰目标。
- 使用 Conventional Commits 中文适配，例如：
  - `feat(游戏): 添加蛇移动和碰撞规则`
  - `feat(认证): 添加用户名密码注册`
  - `test(排行): 覆盖最高分原子更新`
  - `docs(课程): 添加 Docker Compose 入门说明`
- 课程里程碑完成后创建对应 Git 标签。

### 20.2 GitHub 学习内容

- 创建远程仓库和设置 `origin`。
- 推送 `main` 和功能分支。
- 创建 Issue 描述学习任务。
- 创建 Pull Request 查看差异、测试状态和审查意见。
- 合并分支并删除已完成分支。
- 创建 Release 并关联版本标签。

实际创建仓库、推送代码或写入 GitHub Secrets 必须获得用户明确授权。

### 20.3 GitHub Actions

3. 使用 PostgreSQL Service Container 运行数据库与 API 集成测试。
4. 构建前端。
5. 验证 Docker 镜像可以构建。

镜像推送工作流默认不自动启用。用户配置火山引擎 CR 地址和 GitHub Secrets 后，再增加手动触发或版本标签触发的发布流程。

## 21. 测试策略

实现遵循测试驱动开发（Test-Driven Development，TDD）：先写能说明预期行为的失败测试，再完成最小实现并重构。

### 21.1 游戏单元测试

- 初始状态。
- 四方向移动。
- 拒绝直接反向。
- 同一步长拒绝第二次转向。
- 吃食物后成长和加分。
- 每 5 个食物提升速度等级。
- 最低移动间隔限制。
- 食物不出现在蛇身。
- 撞墙和自身碰撞。
- 暂停、继续、重开和胜利。

### 21.2 服务单元测试

- 用户名规范化和校验。
- 密码长度校验。
- 分数范围、倍数和持续时间校验。
- 最高分只增不减。
- 访客最高分正常读写及存储异常降级。
- HTTP 错误到稳定错误码的映射。

### 21.3 API 与数据库集成测试

- 注册成功、重复用户名和非法输入。
- 密码保存为哈希而非明文。
- 登录成功、错误凭据和限流。
- 登录后 Session ID 更新。
- 退出后 Session 失效。
- 未登录不能提交分数。
- 新高分更新，低分和同分不更新。
- 排行榜排序、前 100 名和当前用户排名。
- 数据库迁移可重复执行且不重复应用。
- 数据库故障时健康检查失败。

### 21.4 安全测试

- Session Cookie 包含预期属性。
- 修改请求缺少自定义头或来源不符时被拒绝。
- 简单内容类型不能修改数据。
- SQL 输入不能改变查询结构。
- 错误响应不泄露堆栈和密码哈希。

### 21.5 构建与容器冒烟测试

- 前端生产构建成功。
- Docker 镜像构建成功。
- Compose 启动后应用和数据库均健康。
- 首页、静态资源和 `/healthz` 返回 HTTP 200。
- 注册、登录、提交分数和排行榜完成一次端到端请求链路。
- 删除并重建应用容器后用户和分数仍存在。

### 21.6 人工验收

- 桌面键盘操作。
- 移动端触控操作。
- 响应式布局。
- 注册、登录、退出和错误提示。
- 终端霓虹效果、焦点样式和文字可读性。

## 23. 发布与运维

### 23.1 镜像版本

- 使用语义化版本，例如 `v1.0.0`。
- 可以额外维护 `latest`，但部署和回滚必须使用明确版本。
- 火山引擎 CR 中只推送自定义应用镜像。
- PostgreSQL 从受信任的官方仓库拉取并固定版本。

### 23.2 公网部署

- 注册和登录公开使用前必须启用 HTTPS。
- HTTPS 可由服务器已有反向代理或云负载均衡提供。
- 应用容器只接受来自受控代理或防火墙允许的流量。
- 数据库不对公网发布端口。
- `PUBLIC_ORIGIN` 必须与真实 HTTPS 来源一致。
- 备份和恢复至少成功演练一次后再保存真实用户数据。

### 23.3 升级与回滚

1. 备份数据库。
2. 拉取新应用版本。
3. 启动新版本并等待迁移和健康检查成功。
4. 验证网页、登录和排行榜。
5. 失败时使用上一明确版本重新创建应用容器。

数据库迁移必须保持向后兼容，确保应用镜像回滚时不会立即因表结构不兼容而失败。

## 25. 验收标准

### 25.1 学习体验

- 零基础学习者可以按课程顺序运行每个阶段。
- 每课包含预期结果、常见错误和练习。
- Git 提交与标签能够对应主要学习里程碑。
- README 能清楚区分 GitHub、镜像仓库和 Docker Volume。

### 25.2 产品功能

- 访客无需登录即可游玩。
- 桌面键盘和移动端触控均可完成标准玩法。
- 用户可以注册、保持登录、退出并重新登录。
- 访客最高分保存在当前浏览器，但不会在登录时自动上传。
- 密码不以明文或快速哈希保存在数据库。
- 登录用户的新高分可以进入排行榜。
- 低分和同分不会覆盖最高分时间。
- 排行榜显示前 100 名和当前用户自己的名次。
- 页面明确说明无法找回密码和休闲防作弊边界。

### 25.3 工程质量

- 单元、API、数据库与安全测试通过。
- 前端生产构建通过。
- CI 可以在全新环境中重复执行。
- Docker 镜像使用非 root 用户运行。
- Compose 可以从空 Volume 初始化数据库并启动健康服务。
- 重建应用容器后用户和最高分仍然存在。
- `.env`、真实密钥和数据库文件不进入 Git 或应用镜像。

### 25.4 部署能力

- 应用镜像可以添加版本、推送至火山引擎 CR 并由服务器拉取。
- 服务器可以通过 Compose 启动应用与 PostgreSQL。
- 健康检查、日志、备份、升级和回滚命令具有中文说明。
- 公网注册登录部署明确要求域名、HTTPS 和受控数据库网络。
