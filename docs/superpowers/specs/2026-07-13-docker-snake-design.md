# Docker Snake 设计规格

## 1. 项目背景

`docker_snake` 是一个面向 Docker 入门学习的网页贪吃蛇项目。开发者在本地编写并测试 TypeScript 源码，使用 Vite 生成静态文件，再通过 Docker 多阶段构建制作只包含 Nginx 与静态文件的运行镜像。

镜像上传至火山引擎镜像仓库（Container Registry，CR）后，由开发者自己的服务器拉取并运行。访问者只通过浏览器打开网站，不接触 Docker、镜像仓库或服务器维护命令。

## 2. 目标

- 完成一个可以在桌面端和移动端浏览器运行的贪吃蛇游戏。
- 展示「源码 → 测试 → 编译产物 → Docker 镜像 → 镜像仓库 → 服务器容器」的完整流程。
- 使用纯游戏引擎隔离规则与界面，使核心逻辑可以自动测试。
- 提供中文学习文档，解释开发、构建、运行、推送、部署、升级和回滚命令。
- 让生产镜像只包含运行网站所需的 Nginx 和 `dist/` 文件。

## 3. 非目标

- 不实现用户注册、登录或账号系统。
- 不实现云端排行榜、数据库或后端 API。
- 不在服务器保存玩家分数或游戏进度。
- 不隐藏浏览器端 JavaScript。编译后的前端代码仍会发送到访问者浏览器。
- 第一版不加入音效、障碍物、皮肤或多人游戏。
- 第一版不覆盖域名购买、ICP备案、TLS 证书和公网防火墙配置。

## 4. 总体架构

```text
开发电脑
  TypeScript 源码
        ↓ Vitest
  通过自动测试
        ↓ Vite
  dist/ 静态文件
        ↓ docker build
  Docker 镜像
        ↓ docker push

火山引擎 CR
  版本化镜像
        ↓ docker pull

开发者服务器
  Nginx 容器
        ↓ HTTP

访问者浏览器
  下载 HTML、CSS 和 JavaScript
        ↓
  在浏览器本地运行游戏
```

服务器只负责分发静态文件，不执行游戏规则。游戏移动、碰撞、计分与最高分存储均发生在访问者浏览器中。

## 5. 技术方案

### 5.1 技术栈

- TypeScript：实现具有类型约束的游戏逻辑。
- Vite：提供开发服务器并生成生产静态文件。
- Vitest：执行游戏引擎与存储模块的单元测试。
- HTML Canvas：绘制棋盘、蛇和食物。
- CSS：实现终端霓虹视觉风格与响应式布局。
- Nginx：在容器中提供生产静态网站。
- Docker Compose：简化本地镜像构建与容器启动。

### 5.2 方案选择

采用「TypeScript + Vite + Canvas」，不使用 React 或 Vue。该方案既能展示真实的前端编译阶段，又不会让框架概念分散 Docker 学习重点。

Docker 使用多阶段构建：

1. Node.js 构建阶段安装锁定依赖、运行测试并生成 `dist/`。
2. Nginx 运行阶段只复制 `dist/` 和 Nginx 配置。

## 6. 游戏规则

### 6.1 棋盘与初始状态

- 棋盘为 `20 × 20` 网格。
- 蛇的初始长度为 4，初始方向向右。
- 棋盘上同时存在 1 个食物。
- 初始速度间隔为 140 ms。

### 6.2 移动与输入

- 支持方向键和 WASD。
- 移动端显示触控方向键。
- 同一游戏帧只接受一次有效转向。
- 蛇不能从当前方向直接转向相反方向。
- 暂停状态下不推进游戏，但仍允许继续和重新开始。
- 浏览器标签页进入后台时自动暂停，避免玩家返回后立即失败。

### 6.3 得分与加速

- 每吃到 1 个食物增加 10 分。
- 每吃到 5 个食物提升 1 个速度等级。
- 每级将移动间隔减少 10 ms。
- 最低移动间隔为 70 ms，达到后不再加速。

### 6.4 结束与胜利

- 蛇头超出棋盘边界时游戏结束。
- 蛇头撞到身体时游戏结束。
- 不使用穿墙规则。
- 当蛇占满棋盘、无法再生成食物时进入胜利状态。
- 游戏结束或胜利后可以重新开始。

## 7. 界面设计

### 7.1 视觉方向

采用「终端霓虹」风格：

- 深绿色至黑色背景。
- 低对比度网格线。
- 绿色发光蛇身。
- 红色或粉色发光食物。
- 等宽字体与终端状态文字。
- 发光效果保持克制，保证文字可读性。

### 7.2 页面布局

采用「聚焦控制台」单栏布局：

1. 顶部显示游戏名称、状态、当前分数、最高分和速度等级。
2. 中部显示响应式正方形 Canvas 棋盘。
3. 底部显示开始或暂停按钮、重新开始按钮和操作说明。
4. 小屏设备额外显示触控方向键。
5. 游戏暂停、结束或胜利时，在棋盘上显示状态提示层。

### 7.3 可用性要求

- 页面可通过键盘或触控完成全部游戏操作。
- Canvas 保持正方形并适应可用视口，不产生横向滚动。
- 按钮具有清晰的文字、焦点样式和禁用状态。
- 非 Canvas 文本区域同步显示游戏状态，避免信息只依赖颜色或画面。
- 浏览器不支持 Canvas 时显示可理解的错误提示。

## 8. 前端组件边界

### 8.1 游戏引擎

游戏引擎是纯 TypeScript 模块，不读取 DOM、Canvas 或 `localStorage`。其职责包括：

- 创建初始游戏状态。
- 验证并应用方向变化。
- 推进一个游戏步长。
- 检测墙壁和自身碰撞。
- 处理吃食物、成长、计分和加速。
- 在空闲单元格中生成食物。
- 判断游戏结束和胜利。

随机数生成器通过参数注入，使食物生成可以稳定测试。

### 8.2 输入控制器

输入控制器负责：

- 将方向键、WASD 和触控按钮映射为统一方向。
- 将暂停、继续和重新开始操作转交给应用控制器。
- 阻止触控方向键导致页面意外滚动。
- 忽略当前状态下无效或重复的输入。

### 8.3 计时循环

计时循环按照当前速度等级安排下一次游戏推进。速度改变后使用新的时间间隔，并确保暂停或结束状态不存在残留计时任务。

### 8.4 Canvas 渲染器

渲染器只读取游戏状态并绘制：

- 背景和网格。
- 蛇头、蛇身与食物。
- 暂停、结束和胜利提示层。

渲染器不修改游戏状态。

### 8.5 最高分存储

最高分模块封装 `localStorage`：

- 使用固定且带版本的键名。
- 读取失败、写入失败或数据非法时回退到内存值。
- 不因隐私模式或存储空间异常中断游戏。

### 8.6 应用控制器

应用控制器连接游戏引擎、输入、计时、渲染和最高分模块。它负责维护页面生命周期以及更新非 Canvas 的状态文字。

## 9. 状态与数据流

游戏状态包含：

- 当前阶段：`ready`、`running`、`paused`、`gameOver` 或 `won`。
- 蛇身体坐标。
- 当前移动方向和待应用方向。
- 食物坐标。
- 当前分数、已吃食物数和速度等级。
- 当前移动间隔。

主要数据流如下：

```text
键盘或触控输入
      ↓
输入控制器
      ↓
应用控制器 ─────→ 暂停、继续、重开
      ↓
游戏引擎推进状态
      ↓
Canvas 渲染器 + 页面状态文字
      ↓
最高分模块在需要时保存新纪录
```

游戏状态只存在于内存中。刷新页面会开始新游戏，但最高分保留在当前浏览器中。

## 10. 异常处理

- `localStorage` 不可用：使用内存最高分，游戏继续运行。
- 存储内容不是合法非负整数：忽略该内容并从 0 开始。
- 无法获得 Canvas 2D 上下文：停止初始化并显示错误说明。
- 快速连续输入：每个游戏步长只采用第一个合法转向。
- 食物随机位置落在蛇身：从空闲单元格列表中选择，不进行无上限重试。
- 棋盘没有空闲单元格：进入胜利状态。
- 页面进入后台：自动暂停并清理活动计时任务。
- 容器健康检查失败：保留失败状态供 Docker 查看，不自动掩盖 Nginx 错误。

## 11. Docker 与 Nginx 设计

### 11.1 构建阶段

- 使用 Node.js 活跃 LTS 的 Alpine 基础镜像。
- 先复制依赖清单和锁文件，再安装依赖，以利用 Docker 构建缓存。
- 使用锁文件安装命令保证依赖可复现。
- 复制源码后运行单元测试和生产构建。
- 任一测试或构建步骤失败时终止镜像构建。

### 11.2 运行阶段

- 使用受信任的 Nginx Alpine 基础镜像。
- 删除默认网站内容并复制 `dist/`。
- 仅暴露 HTTP 端口 80。
- 提供 `/healthz` 健康检查端点。
- 为带内容哈希的静态资源设置长期缓存。
- 为 `index.html` 设置短期或禁用缓存，便于版本更新。
- 添加适合纯静态网站的基础安全响应头。
- 添加 Docker `HEALTHCHECK`。

### 11.3 构建上下文

`.dockerignore` 排除：

- `node_modules/`
- `dist/`
- Git 元数据
- 测试覆盖率文件
- 编辑器配置和本地日志
- `.superpowers/` 临时视觉原型

## 12. 测试策略

### 12.1 单元测试

至少覆盖：

- 创建初始状态。
- 四个方向移动。
- 拒绝直接反向。
- 同一游戏步长拒绝第二次转向。
- 吃食物后成长和加分。
- 每 5 个食物提升速度等级。
- 速度不会超过设定上限。
- 食物不会生成在蛇身上。
- 撞墙进入游戏结束状态。
- 撞到自身进入游戏结束状态。
- 棋盘填满后进入胜利状态。
- 暂停、继续和重新开始状态转换。
- 最高分正常读写以及异常降级。

### 12.2 构建验证

- TypeScript 类型检查通过。
- Vitest 全部通过。
- Vite 生产构建成功。
- `dist/` 包含入口 HTML 与版本化静态资源。

### 12.3 容器冒烟测试

- Docker 镜像构建成功。
- 容器可以启动并达到健康状态。
- 网站首页返回 HTTP 200。
- `/healthz` 返回 HTTP 200。
- 首页可以引用并下载生产 JavaScript 和 CSS。
- Nginx 日志中不存在启动错误。

## 13. 发布与运维

### 13.1 镜像命名

镜像使用语义化版本标签，例如 `v1.0.0`。可以额外维护 `latest`，但部署和回滚命令必须使用明确版本。

### 13.2 火山引擎 CR 参数

README 将把下列值定义为部署参数，而不是写入源码：

- 镜像仓库域名。
- 命名空间。
- 仓库名称。
- 镜像版本。

登录凭证只传给 `docker login`，不写入 Dockerfile、Git 或镜像层。

### 13.3 服务器部署

服务器部署流程：

1. 登录火山引擎 CR。
2. 拉取指定版本镜像。
3. 启动新版本容器并检查健康状态。
4. 确认网页可以访问后移除旧容器。
5. 保留上一稳定版本号，以便重新创建旧版本容器。

本地默认将宿主机端口 8080 映射到容器端口 80。公网服务器可直接映射端口 80，或只绑定本机端口并交给已有反向代理提供域名和 HTTPS。

### 13.4 日常维护

README 覆盖以下操作：

- 查看容器状态和健康状态。
- 查看并跟随 Nginx 日志。
- 停止、启动和重新创建容器。
- 拉取并切换到新版本。
- 回滚至上一明确版本。
- 清理无引用的旧镜像。

玩家最高分只存放在各自浏览器中，因此服务器升级、替换容器或删除镜像不会迁移或删除玩家浏览器中的最高分。

## 15. 验收标准

- 用户可通过服务器 URL 打开贪吃蛇游戏，无需安装任何软件。
- 桌面键盘和移动端触控均可完成标准玩法。
- 暂停、重新开始、加速、最高分、结束和胜利状态工作正常。
- 全部自动测试与生产构建通过。
- Docker 镜像可以构建、启动并通过健康检查。
- 最终运行镜像不包含 Node.js 运行环境或开发依赖。
- README 能指导初学者完成本地开发、镜像构建、火山引擎 CR 推送、服务器拉取、升级和回滚。
