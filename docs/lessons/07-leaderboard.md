# 第 7 课：最高分与休闲排行榜

本项目只保存每个用户的最高分，不保存每一局历史记录。以后需要对局历史时，可以新增独立的 `game_runs` 表。

## 提交成绩

浏览器只发送：

```json
{
  "score": 120,
  "durationMs": 840
}
```

服务器检查：

- 对象只能包含这两个字段；
- 分数为 0–3960 的整数和 10 的倍数；
- 时长为 0–86400000 ms 的整数；
- 正分时长不能短于理论最快移动时间；
- 请求必须已经登录。

服务器使用一条带条件的 `UPDATE`。只有新分数更高时才修改最高分和达到时间，因此两个并发请求也不会用低分覆盖高分。

## 排名规则

排行榜只包含正分用户，最多返回 100 人，排序顺序为：

1. 最高分从高到低；
2. 同分时，较早达到该分的人在前；
3. 时间仍相同时，用户 ID 较小的人在前。

个人排名使用完整正分用户集合计算。因此即使用户排在第 105 名，页面仍能显示真实的 `#105`，而不是误认为没有排名。

## 每局归属为什么在开始时冻结

如果只在游戏结束时读取当前登录用户，会出现串分：

```text
A 开始游戏 → 中途退出并登录 B → 游戏结束 → 分数错误提交给 B
```

`score-coordinator.js` 在开始或重开时生成冻结 token：

```js
{ kind: 'user', userId: '1' }
```

结束和手动重试始终复用同一个 token。只有原用户仍然登录时才会提交。游客开局后即使中途登录，成绩也只写入游客本地存储。

## 防止旧排行榜覆盖新状态

排行榜刷新使用两层保护：

- 新请求通过 `AbortController` 中止旧请求；
- generation 编号确保晚到的旧响应无法更新 DOM。

用户名使用 `textContent` 写入页面，不使用 `innerHTML`，避免把用户名当成 HTML 执行。

## 本课练习

1. 用两个账户提交不同分数，观察排行榜顺序；
2. 同一账户先提交高分再提交低分，确认最高分不下降；
3. 游客开局后登录，确认游客分没有上传；
4. 运行 `node --test tests/client/score-coordinator.test.mjs tests/client/leaderboard.test.mjs`。
